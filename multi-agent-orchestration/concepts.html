1) Multi‑Agent Systems: the core idea

The core shift is **teamwork**: multiple specialized agents behave like a production team (planning, execution, review, retrieval) instead of one “do everything” agent.

Example team

  • Planner: decomposes goals into steps
  • Executor: runs tools and transforms results
  • Reviewer: validates, blocks unsafe actions
  • Retriever: queries corpora/DBs/web

Why this matters

  • Separates planning vs execution vs validation
  • Reduces hallucinated tool outputs
  • Enables guardrails + budgets per role
Architect mindset: you are designing **roles, contracts, and failure recovery** — not prompts.

2) Role‑Based Agents (specialization)

Role separation prevents one agent from doing long reasoning and fragile execution in the same loop. 

User request: “Find and summarize recent papers on quantum computing”

Planner:
  1) Retriever → search academic sources
  2) Executor  → download top papers
  3) Executor  → extract key findings
  4) Reviewer  → verify citations + sanity-check claims
Production trick: keep each role’s context scoped. Give the executor only what it needs to execute the current step.

3) Tool Use with Validation

Tools fail. Your system must be resilient: validate inputs, enforce timeouts, validate outputs, retry with backoff, and fall back safely.

Bad (no validation)

def get_weather(city):
  return api.call(city)  # may timeout / return errors / wrong schema

Good (validation + timeout)

Show robust wrapper example 
def get_weather(city: str) -> dict:
  if not isinstance(city, str) or len(city) < 2:
    raise ValueError("Invalid city")
  try:
    resp = api.call(city, timeout=5)
  except Timeout:
    return {"error":"timeout","retry":True}
  if "temp" not in resp or "conditions" not in resp:
    raise SchemaError("Missing fields")
  return resp
Tool reliability is where most agent demos die. Make tool wrappers boring, strict, and well-tested.

4) Stateful Execution (memory & context)

Multi-step work needs state, and production needs resume/replay. Use short-term run state plus governed long-term memory.

Short-term (in-run)

{
  "current_step": 3,
  "tool_results": { "search_flights": {"found": 5} },
  "context": "User prefers morning flights"
}

Long-term (cross-run)

{
  "user_preferences": {
    "preferred_airlines": ["Delta","United"],
    "last_booking": "2026-01-15"
  }
}
Memory without governance creates drift. Treat memory like a database: schemas, retention, and versioning.

5) Budgets & termination guarantees

Without explicit budgets, agents can loop forever. Budgets are how you guarantee termination and control cost.

  • Token budget across all model calls
  • Time budget per run (wall-clock)
  • Tool-call budget per run
  • Cost budget (USD)
  • Clarification cap to prevent dialogue loops
# Tool call budget (sketch)
max_tool_calls = 20
tool_calls = 0
def call_tool(name, args):
  global tool_calls
  if tool_calls >= max_tool_calls:
    terminate("tool_budget_exceeded")
  tool_calls += 1
  return execute_tool(name, args)

6) Observability (tracing & logging)

Production debugging requires step-by-step traces: what agent decided, which tool ran, inputs/outputs, latency, and cost.

Example trace shape 
{
  "run_id":"run_abc123",
  "steps":[
    {"step_id":1,"agent":"planner","action":"create_plan","duration_ms":450},
    {"step_id":2,"agent":"executor","action":"tool_call","tool":"db_query","status":"success"}
  ],
  "final_status":"success",
  "total_duration_ms":1739
}
If you can’t replay a failure deterministically, you don’t control the system.

7) Failure modes (and how systems recover)

The most common failure modes are non-termination loops, state drift, and tool brittleness. Your orchestrator must handle them explicitly.

Non‑termination loop

Planner: fetch user data
Executor: need clarification
Planner: fetch user data
Executor: need clarification
...

Fix

  • max clarification rounds
  • max retries per tool
  • hard max steps per run
Reliability is “boring engineering”: caps, timeouts, schemas, checkpoints, and fallbacks.

8) Security threats (prompt injection, tool injection, exfiltration)

Assume inputs and tool outputs may be malicious. Security is about enforcing contracts and policies — not trusting text.

  • Prompt injection: user tries to override policies
  • Tool injection: tool output includes “instructions” fields
  • Exfiltration: sensitive data leaking via tools/logs

Defense pattern

  • tool allowlists per agent/run
  • schema-based parsing (ignore extra fields)
  • redaction before logs/artifacts
  • approval gates for destructive tools

Where it lives

  • policies/ (allow/deny)
  • schemas/ (input/output validation)
  • redaction/ (log safety)
  • audit/ (who/what/when)

9) Execution graph (DAG / state machine)

Use an explicit execution graph: it makes retries, budgets, and replay deterministic and debuggable. 

Intake → Plan → Dispatch → Tool Run → Review → Commit → Terminate
If your orchestration is “just recursive prompting”, you can’t guarantee termination or safety.

10) Contracts (Tool schema + Agent interface)

Contracts reduce ambiguity. Schema validation is your primary defense against brittle tools and injection via unexpected fields.

Tool contract

  • name + description
  • input_schema + output_schema
  • timeout + retry policy
  • side effects + safety level

Agent contract

  • action: tool_call/delegate/finalize
  • structured tool payload
  • state updates (patches)
  • final artifacts/citations

11) Checkpointing (resume + replay)

Checkpoints are how you resume after crashes and replay failures deterministically.

  • checkpoint after each step (atomic commit)
  • store tool IO and agent outputs as artifacts
  • support replay for regression testing

12) Evaluation harness (prove it works)

Your system isn’t “production-ready” until you can measure reliability, cost, and quality over a task suite.

  • start with 20–50 tasks
  • run each task multiple times (stochasticity)
  • track success rate, p95 latency, cost, retries, safety catches
  • store results for comparisons between versions
Evaluation is not a report. It’s a guardrail that prevents regressions and keeps costs bounded.
